# effort-estimate スキル設計

## 目的

特定タスク1本の工数（人日）を、日報・gitログ・Rocket Chat・Claudeセッション要約の一次データから逆算して見積もる。「〇〇にどれくらい時間がかかったか」を精度高く振り返るためのスキル。

`work-report`（期間まるごと→カテゴリ別箇条書き）とは目的・集計軸が直交するため、別スキルとして分離する。

## スコープ

### やること
- タスク名テキストを入力に、関連する作業期間を全ソース横断で発見する
- 期間内のデータを収集し、並行作業を除外して人日ベースで工数を算出する
- 調査メモ形式でチャットに出力する

### やらないこと（YAGNI）
- ファイル保存・git commit（`work-report` と同じくチャット出力のみ。保存はユーザーが指示）
- 期間まるごとのカテゴリ別報告（それは `work-report` の役割）
- KPT・生産性指標などの追加集計

## アーキテクチャ

- 新規スキル `effort-estimate` を `.claude-global/skills/effort-estimate/SKILL.md` に作成
- データソース定義（gh CLI・gitログ・Rocket Chat・Claudeセッション要約の取得コマンド）は `work-report` からコピーして流用する
  - 将来共通化の余地はあるが、現時点では各スキル自己完結でよい（2ファイルの重複は許容）

## 処理フロー

1. **入力**: タスク名テキストのみ（例「タカショーのデモ環境作成」）
2. **期間発見**: 全ソース（blog md・Claudeセッション要約・各gitリポジトリ）をタスクのキーワードでgrep横断し、ヒットした日付範囲を当たりとして特定する
3. **境界確認**: 含める範囲の候補が複数あり一意に決まらないときだけ `AskUserQuestion` で確認。それ以外は広めに取る（迷ったら出す側）
4. **データ収集**: 期間内の日報サマリー・gitログ（タイムスタンプ付き）・Claudeセッション（start/end付き）・Rocket Chat を取得
5. **工数算出**: 下記 methodology に従う
6. **出力**: 調査メモ形式でチャットに出力（保存なし）

## 工数算出の methodology（スキルの肝・決定的手順として明文化）

セッション時間の単純合算は破綻するため、以下を必ず守る：

1. **人日ベースが主軸**: 日報の申告を一次ソースにする
2. **セッションの start/end は合算しない**: 同一 session_id の重複要約・複数日またぎのコンテキストにより、合算すると実作業時間の2〜3倍に膨らむ。commit タイムスタンプとセッション時刻は「その日の作業窓の下限」を裏取りするクロスチェックにだけ使う
3. **並行作業を各日から除外**: そのタスクに無関係な別リポジトリ・別機能・個人リポの作業を各日の人日から差し引く
4. **レンジで出す**: 「37.5h」のような偽精度は禁止。「約6人日」「1人日 ≒ 7.5h 換算」のように推定であることを明示する
5. **桁のサニティチェック**: 肌感とオーダーがずれたら二重計上を疑う（例：一連の弧が肌感5〜6人日なのに30hや150hが出たら方法を疑う）

## 出力形式（調査メモ・デフォルト）

- 合計工数（レンジ）を先頭に置く
- 作業カテゴリ別の内訳＋各人日
- 日別クロスチェック表（作業窓 JST・並行除外作業）
- 前提の明記（推定である旨・除外したもの・換算レート）
- 上司報告用の簡潔版が必要なら、ユーザーが追加指示 → 詳細版から削って作る

## エラー処理

`work-report` に準ずる：

| 状況 | 対応 |
|---|---|
| タスク名でヒットするデータが1件もない | 「該当する作業が見つかりません」と表示して終了 |
| gitリポジトリなし | スキップして他ソースで算出 |
| Rocket Chat 発言0件 | スキップして他ソースで算出 |
| 日報が期間内に1日もない | gitログ・セッション要約のみで算出（精度低下を明記） |
