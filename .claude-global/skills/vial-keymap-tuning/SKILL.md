# Vialキーマップ調整

Vial（`get.vial.today`）はQMKファームウェア上で動くGUIコンフィギュレータ。レイヤー・タップホールド・コンボ・QMK Settingsをすべて`.vil`ファイル（JSON）で管理する。このスキルは主に「レイヤーキーのタップホールド判定の誤動作」の診断と、キーマップ全体のレビュー・改善提案を行う。

マクロ機能（`macro`フィールド）は対象外。

## 前提知識: Vialの設定構造

`.vil`ファイルはJSONで、主要フィールドは以下:

| フィールド | 内容 |
|---|---|
| `layout` | レイヤーごとのキー配列（`layer[row][col]`） |
| `encoder_layout` | ロータリーエンコーダーの割り当て |
| `macro` | マクロ定義（本スキルでは扱わない） |
| `tap_dance` | Tap Dance定義（`[tap, hold, doubletap, taphold, tappingterm]`） |
| `combo` | コンボ定義 |
| `key_override` | キーオーバーライド |
| `settings` | QMK Settingsタブの値。**キーは数値QSID**でVial GUI側のマッピングに依存するため、`.vil`を直接読んでも項目名は分からない。必ずVial GUIの「QMK Settings」タブを開いて確認・変更すること |

`.vil`ファイルの直接編集は非推奨。Vial GUI（Design tab / QMK Settings tab）経由での変更が安全（GUI側でVIA/QMKプロトコルとの整合性を保証しているため）。

## 診断: レイヤーキーが高速入力時に発動しない問題

### 症状
`LT(layer, kc)`（Layer Tap）や `MT(mod, kc)` で数字レイヤー等を呼び出しているキーで、パスワード入力のように高速に連打すると、ホールドとして判定されずタップ（ベースレイヤーの文字）が送出されてしまう。

### 根本原因（QMK標準のタップホールド判定ロジック）

QMKのデフォルト動作は次の通り:
1. タップホールドキーを押してから `TAPPING_TERM`（デフォルト200ms）以内に離せば「タップ」
2. `TAPPING_TERM`を超えて押し続ければ「ホールド」
3. **デフォルトでは、タッピングターム内に別のキーが押されても（ネストでもロールでも）ホールドは発火せずタップとして処理される** — これが高速タイピング時に「レイヤーが乗らない」主因

このデフォルト挙動は、元々ホームロー修飾キー（Shift/Ctrl等をアルファベットキーに重ねる運用）で「タイピング中にたまたま重なっただけの押下」を誤ってModとして発火させないための保守的な設計。レイヤーキーとして能動的に使う場合は「押している間に他キーが来たら即レイヤーに行くべき」が自然な期待であり、これは後述の**Hold On Other Key Press**がそのまま実現する。デフォルトのままだとこの直感に反する挙動になる点が今回の症状の本質。

対象リポジトリの`cornix.vil`では、数字レイヤーが `LT2(KC_LANG2)` / `LT3(KC_LANG1)` のようにLayer Tapで呼ばれており、上記の挙動がそのまま起こりうる構成になっている。

### 改善策（QMK Settings / Vial GUIで設定）

Vial GUIの「QMK Settings」→「Tap-Hold」カテゴリで以下を調整する。設定は個別キー単位ではなくグローバル（全タップホールドキー共通）である点に注意。

| 設定 | 効果 | 高速タイピングへの向き不向き |
|---|---|---|
| **Hold On Other Key Press** | タッピングターム内でも、他のキーが押された瞬間に即座にホールド判定にする | 最も強力。高速タイピスト向け。ただしロール入力（隣接キーを連続で叩くタイプ運指）で誤ホールドしやすい副作用がある |
| **Permissive Hold** | 他キーが「押されて、かつ離される」までタッピングターム内に完結した場合にホールド判定にする（ネスト押下のみ対応、ロールには非対応） | Hold On Other Key Pressよりは誤爆が少ないが、ロール入力にはやや弱い |
| **Retro Tapping** | タッピングターム超過後に他キーを挟まず離した場合、ホールドではなく元のタップキーコードを送出する | ホールド誤発火の反対（タップが欲しいのにホールドになるケース）向けの設定なので今回の問題には直接効かない |
| **Tapping Term** | タップ/ホールドの判定時間そのもの（デフォルト200ms） | 短くしすぎるとホールドに倒れやすくなるが、意図しないホールドが増える諸刃の剣。まずは200ms前後を維持し、Hold On Other Key PressかPermissive Holdで対処するのが定石 |
| **Chordal Hold**（対応ファームウェアのみ） | 「反対の手」ルール。タップホールドキーと同じ手のキーが押された場合はホールドにせずタップ扱いにする | 分割キーボードで左右の手をまたぐ運指の誤爆を抑えたい場合に有効。ロール入力との相性が良い |

### Tapping Termの目安

- デフォルトは200ms。**50ms以下など極端に短い値は、通常のタイピングでもホールドに倒れやすくなり逆効果**（タップしたい文字が化ける）
- 高速タイピスト向けの目安は **150〜180ms**。短くしすぎず、Hold On Other Key Press / Permissive Holdと併用するのが定石
- Tapping Termは「時間」のグローバルしきい値でしかなく、「他キーが押された」というイベントには反応しない。高速連打時の重ね押し問題（今回のような症状）にはHold On Other Key Press等の設定の方が直接効く

### 推奨の当てはめ方

1. まず **Hold On Other Key Press** を有効化して様子を見る（最も直感的にレイヤーが乗るようになる）
2. ロール入力（例: 連続する文字を素早く打つ癖がある）で誤ホールドが増えるようなら **Permissive Hold** に切り替える
3. 分割キーボードで「反対の手」判定が使えるファームウェアなら **Chordal Hold** も検討する
4. パスワード欄など数字レイヤーの用途が決まっているキーだけを個別に強くしたい場合は、`LT()`ではなく `MO(layer)` を別の指の専用キーに割り当てる設計に変更する方法もある（タップホールド判定自体を回避できる）

### 診断時にユーザーへ確認すること

- どのレイヤーキーで問題が起きているか（キー位置・現在のキーコード）
- 高速入力時の運指パターン（ロール型か、単純な連打か）
- 分割キーボードかどうか（Chordal Holdが使えるファームウェアか）
- 現在のTapping Term設定値（Vial GUIのQMK Settingsタブで確認してもらう）

## キーマップ全体レビューの観点

「作りたて」のキーマップをレビューする際にチェックする項目:

1. **ホームロー・頻出キーの到達性**: よく使う記号・修飾キーがホームポジションから遠くないか
2. **同一指連続（SFB: Same Finger Bigram）**: レイヤー変更で頻出する2打鍵が同じ指に連続で当たっていないか
3. **修飾キーの配置**: Shift/Ctrl/Altが親指またはホームロー下段など、押しながら他キーを打ちやすい位置にあるか
4. **レイヤー設計の一貫性**: 数字・記号・ナビ・ファンクションなど用途別レイヤーの配置が体系的か（例: 数字レイヤーと同じ位置に対応する記号があるか）
5. **未使用スロットの活用**: `KC_NO`が並ぶ空きキーに、コンボやTap Danceで機能を足せる余地がないか
6. **タップホールドの多用しすぎに注意**: `LT()`/`MT()`を多用すると本スキルで扱う誤発火リスクが増える。本当に必要な箇所に絞る
7. **IME切り替えキーの位置**: 日本語入力を使う場合、`KC_LANG1`/`KC_LANG2`が誤爆しにくい位置にあるか（頻出キーの近くに置くと英数/かな切り替えが暴発しやすい）

レビュー時は `.vil`ファイルの`layout`配列を読み、各レイヤーのキーコードを一覧化してから上記観点で指摘する。
