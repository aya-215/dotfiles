# PR対話レビュー

PRをClaudeと一緒に読み、コードの理解を深めるためのスキル。
Claudeが能動的に進行役となり、各フェーズで必ず理解確認を行う。

## 引数

$ARGUMENTS

- 引数なし: **現在ブランチに紐づくPRを対象にする**（デフォルト）
- PR番号のみ: `123`
- PR番号＋リポジトリ: `123 owner/repo`

## Iron Law

```
理解確認なしに次のフェーズへ進まない。
AskUserQuestionで確認を取るまで、絶対に先へ進むな。
```

## AskUserQuestion Rules

- **1メッセージ = 1質問のみ**
- 毎フェーズ終了後に必ずAskUserQuestionで確認する
- ユーザーが「次に進む」を明示的に選択するまで、絶対に次のフェーズへ進まない
- ユーザーが質問・深掘りを選んだ場合 → その質問に答えた後、**再度AskUserQuestionを出して確認を取る**（自動で次へ進まない）
- AskUserQuestionの選択肢には必ず「次に進む」という選択肢を含める
- **例外**: ユーザーが自分の論点・疑問を持ち込んだときは、AskUserQuestionを挟まずまず追従する。議論・検証・意見表明の途中に確認を差し込まない（詳細は「検証と対話のバランス」節）。一段落してから確認フェーズに戻す

## The Process

### Step 0: PR特定

優先順位: **明示指定 > 現在ブランチ > 聞く**

1. 引数でPR番号が指定されていればそれを使う
2. 引数なしの場合、`gh pr view`（番号なし）で**現在ブランチに紐づくPRをそのまま対象にする**。確認のAskUserQuestionは出さない
3. 現在ブランチにPRが無く `gh pr view` が失敗したときだけ、AskUserQuestionでPR番号を聞く：

```
「現在ブランチにPRが見つからへんかった。どのPRをレビューする？PR番号を教えて（リポジトリが違う場合は owner/repo も）」
```

### Step 1: PR情報取得

```bash
# 引数なし（現在ブランチのPR）
gh pr view
gh pr diff

# PR番号指定の場合
gh pr view <PR番号>
gh pr diff <PR番号>

# リポジトリを指定する場合
gh pr view <PR番号> --repo <owner>/<repo>
gh pr diff <PR番号> --repo <owner>/<repo>
```

### Step 1.5: hunk連携（オプション）

PR情報を取得したら、hunkのTUIで視覚的にレビューするか確認する。

→ **[AskUserQuestion]** 「hunkのTUIで差分を見ながらレビューする？それともテキストで進める？」
- 選択肢: 「hunkで見る」/「テキストで進める（デフォルト）」

**「hunkで見る」が選ばれた場合のみ:**

1. 連携手順は `references/hunk-session.md` に従う（**必ず読むこと**）
2. **Claudeは `hunk diff` を自分で実行しない。** ユーザーに開いてもらう:
   「`hunk diff <base>...HEAD` を自分の端末で開いてな（`<base>` はPRのマージ先）」
   （ローカルにPRブランチがcheckout済みの前提。未checkoutなら `gh pr checkout <番号>` を案内）
3. 「開いた」と言われたら `hunk session list` で接続
4. 以降のStep 3（設計判断の解説）で、設計判断を説明するたびに
   `hunk session navigate` で該当hunkへ画面を動かし、所見を `hunk session comment add` で添える
5. `navigate` は自由・`comment` は付けたら報告・`clear`/`rm` は承認必須

**「テキストで進める」が選ばれた場合:**
従来通り `gh pr diff` のテキストベースで進行する（hunkは使わない）。

### Step 2: 全体概要の説明

以下を簡潔に説明する（箇条書き、3〜5行）：
- このPRが「なぜ」作られたか（背景・目的）
- 変更されたファイルの一覧
- 変更の大まかな規模感

→ **[AskUserQuestion]** 「全体像はつかめた？気になる点や先に聞きたいことはある？」

### Step 3: 設計判断・実装意図の解説

**重心は「コードが何をしているか」ではなく「なぜこう作ったか」に置く。** ユーザーが最も知りたいのは実装の意図・設計判断であることが多い。

変更を**設計判断の単位**で取り上げ、以下を中心に説明する：
1. **なぜこの設計・実装を選んだか**（意図・背景）
2. **他にどんな選択肢があり、なぜそれを採らなかったか**（トレードオフ）
3. その判断が妥当か、過剰／不足はないか

**設計判断はファイル境界に縛られず横断的に扱う。** 1つの設計判断が複数ファイルにまたがることは多い（例: ツール層＋サービス層＋登録集約点）。ファイル単位で区切らず、意図の単位でまとめる。

コードの逐次的な仕組み解説は、**意図を理解するのに必要な範囲に絞る**。「このファイルを1行ずつ丁寧に」は必須ではない。ユーザーが「このファイル詳しく」と求めたときだけ深掘りする（オプション）。

→ **[AskUserQuestion]** 「この設計判断、意図は伝わった？気になる点や深掘りしたい所はある？」

設計判断の数だけこのステップを繰り返す。次の論点に進む前に必ず確認する。

### Step 4: 影響範囲の説明

- 他のコードへの波及
- 外部サービスやAPIへの影響
- 破壊的変更の有無

→ **[AskUserQuestion]** 「影響範囲についてはどう？気になる点はある？」

### Step 4.5: 影響コード追跡

変更されたコードの呼び出し元・依存先を追跡する：
- 変更された関数/コンポーネントを使用している箇所をgrepで検索
- 型変更がある場合、影響を受ける型の利用箇所を確認
- APIの変更がある場合、クライアント側への影響を確認

影響が見つかった場合はユーザーに報告し、Step 5のレビュー観点に含める。
影響がない場合はその旨を伝えて次へ進む。

### 検証ガード（所見を出す前に必須）

ユーザーの疑問への回答・レビュー所見・設計上の指摘を述べる前に、必ず一次情報で裏取りする：

1. 前提（設計書・引き継ぎ・**ユーザーの推測・自分の直感・相手への同調**）を鵜呑みにしない
2. 本体コード・実データ・実挙動で確認してから結論を出す
3. 各所見を **CONFIRMED / PLAUSIBLE / REFUTED** に分類して提示する
   - CONFIRMED: 一次情報で確認済み
   - PLAUSIBLE: もっともらしいが未確認（「まだ未確認だが」と明示）
   - REFUTED: 確認したら誤りだった

裏取り前の推測を先に述べない。特に、ユーザーの疑問に反射的に同調して未検証の所見を出さない（「ご指摘の通り」の前に確認する）。

### 検証と対話のバランス

検証ガードで確認を増やしすぎないための運用ルール：

- **裏取り（コード確認・実データ検証・advisor相談）は AskUserQuestion で許可を求めず、その場で実行してよい。** 検証のたびに確認を挟まない
- ユーザーが自分の論点・疑問を持ち込んだら、フェーズ確認を挟まず**まずそれに追従する**
- **議論・検証・意見表明の途中では、確認を差し込まない。** 追従の一連（裏取り→分類付き所見の提示）が完了してから、確認フェーズに戻す
- 論点が一段落したら、元のフェーズ進行に戻す（追従したうえで戻す）

### Step 5: レビュー観点の整理

以下のチェックリストに基づき、変更内容を評価する：

**チェック観点:**
- セキュリティ: 入力検証、認証・認可、シークレット漏洩
- パフォーマンス: N+1、不要な再計算、メモリリーク
- エッジケース: null/undefined、空配列、境界値、並行処理
- テスト: 変更に対するテストの有無、異常系カバレッジ
- 互換性: 破壊的変更、既存APIへの影響

**重要度分類:**
- Critical: マージすべきでない問題
- Important: マージ前に対応が望ましい問題
- Nit: 改善提案（マージには影響しない）

問題が見つかった場合は重要度を付けて提示する。
問題がない場合は「確認した結果、特に問題は見つからなかった」と明示する。
良い点（意図的な設計判断、コード品質の向上等）も挙げる。

→ **[AskUserQuestion]** 「レビュー観点として気になるものはあった？他に確認したいことは？」

### Step 6: レビューサマリー

対話を通じて得られた内容を以下の形式で整理する：

1. **良かった点**: 対話中にユーザーが評価した点 + Step 5の良い点
2. **確認した点**: Step 3-4で理解確認できた内容
3. **気になった点・質問**: Step 5の問題点（重要度付き）
4. **レビュー判定の推奨**: Approve / Request Changes + 理由

→ **[AskUserQuestion]** 「サマリーを整理した。修正する点はある？このままレビューコメントを書くなら /write-review-comment を使ってね」

## gh CLI Reference

```bash
# PR概要（番号省略＝現在ブランチのPR）
gh pr view
gh pr view <PR番号>

# diff（全体）
gh pr diff
gh pr diff <PR番号>

# 変更ファイル名のみ
gh pr diff --name-only
```

認証エラーが出た場合は `gh auth switch` でアカウントを切り替える。
