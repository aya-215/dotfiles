# 上司向けトーンガイド

## 基本原則

- 設計判断を「評価する」のではなく「理解した・学びになった」という姿勢で書く
- 自分が何を確認したかを明記する（ちゃんとレビューした証拠）
- 問題指摘は「〜ご確認いただきたいです」系の依頼形を基本にする。断定・命令形（「〜してください」「〜すべき」）は使わない
- 良い点を必ず含める（改善点だけだとネガティブ）

## 冒頭の挨拶

固定表現ではなく、状況に応じて選ぶ:

| 場面 | 表現 |
|------|------|
| Approve（推奨デフォルト） | 「実装ありがとうございます！」 |
| Approve（シンプル） | 「確認しました！」 |
| Request Changes | 「確認しました！」 |

## 表現変換テーブル

### 評価

| 避ける表現 | 使う表現 |
|---|---|
| 「〜が良い」「〜が適切」 | 「〜だと思いました」「〜が勉強になりました」 |
| 「設計が優れている」 | 「〜という設計方針、今後の拡張にも対応しやすいと思いました」 |
| 「正しい判断です」 | 「〜の判断は妥当だと理解しました」 |
| 「よく考えられている」 | 「〜の観点が考慮されている点、参考になりました」 |

### 確認事項・質問

| 避ける表現 | 使う表現 |
|---|---|
| 「〜済みか？」「〜すべき」 | 「〜予定はありますか？」「〜気になりました」 |
| 「〜を確認してください」 | 「〜について確認させてください」 |
| 「なぜ〜ですか？」 | 「〜の背景を教えていただけますか？」 |

### 問題指摘（Request Changes時）

問題指摘の見出し・依頼文は「〜ご確認いただきたいです」系の依頼形を基本にする。

| 避ける表現 | 使う表現 |
|---|---|
| 「〜が間違っています」 | 「〜のように見えるのですが、意図的でしょうか？」 |
| 「〜を修正してください」 | 「〜ご確認いただきたいです」 |
| 「〜が必要です」「〜すべきです」 | 「〜だけご確認いただきたいです」 |
| 「バグがあります」 | 「〜の挙動が想定と異なるように見えました」 |
| 「テストが不足しています」 | 「〜のケースもテストがあると安心かと思いました」 |

修正依頼が1点のみの場合、本題の前に軽い前置きを入れると角が立ちにくい:
- 「〜だけご確認いただきたいです」
- 「〜の部分だけ気になったので共有させてください」

### 自分の確認内容

以下のパターンで「何を確認したか」を具体的に記述する：

- 「〜が追従していることを確認しました」
- 「テストで〜がカバーされていることを確認しました」
- 「〜の動作を確認しました」

## Approve時の構造

```markdown
実装ありがとうございます！

良いと感じた点
- （変更前→変更後の構造変化に言及 + 「〜だと思いました」形式）

確認した点
- （Step 2 でユーザーが申告した内容のみ。申告がなければセクションごと省略）

質問（マージには影響しません）
- （質問形式。なければセクション自体省略）
```

## Request Changes時の構造

```markdown
確認しました！

良いと感じた点
- （良い点を必ず先に書く）

ご確認いただきたい点
- （「〜ご確認いただきたいです」系の依頼形で問題を提示 + 理由を添える）

質問
- （理解のための確認事項）
```
